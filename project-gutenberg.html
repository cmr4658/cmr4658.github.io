Project Gutenberg

UI Redesign

Project Overview

Project Gutenberg is an online library of free, public domain eBooks. It's an important resource for literature lovers around the world (and a website I use frequently!) but there are some areas where the user experience could be improved. My hope is that redesigning the interface could make it easier for users to understand what Project Gutenberg is about and find the books they want to read.

Client

Prof. Daniel Kramer (Course Instructor)

Team

Chase Rigby (Designer)

Timeline

8 weeks, 3-6 hours/week

Process

  1. Heuristic evaluation
  2. Low-fidelity prototyping
  3. High-fidelity prototyping
  4. Final design

Heuristic Evaluation

I used Jakob Nielsen's 10 Usability Heuristics to evaluate the Project Gutenberg website. These were the heuristic violations I decided to address in the redesign:

Recognition rather than recall

Minimize the user's memory load by making elements, actions, and options visible. The user should not have to remember information from one part of the interface to another. Information required to use the design (e.g. field labels or menu items) should be visible or easily retrievable when needed.

Project Gutenberg is (naturally) a very text-heavy website, which isn't a bad thing. However, the amount of text on the home page might make it difficult for users to find frequently-used actions (like browsing and advanced search) or even to understand what they can do on the website.

Next steps

Reorder the content on the home page so important and frequently-used options are near the top. Also, remove extraneous text to keep the page focused.

Consistency and standards

Users should not have to wonder whether different words, situations, or actions mean the same thing. Follow platform and industry conventions.

Project Gutenberg is mostly consistent between pages, but the horizontal layout of the search results page sticks out for two reasons:

  • It looks different from any other page on the website (i.e., it lacks internal consistency)
  • It looks different than most other library search results pages (i.e., it lacks external consistency)

Next steps

Changing the search results page to a vertical layout would make it easier for new users to learn the interface.

Help and documentation

Help and documentation content should be easy to search and focused on the user's task. Keep it concise, and list concrete steps that need to be carried out.

There is actually quite a bit of documentation available on Project Gutenberg's website, but it's spread across dozens of pages, and it can be difficult to search. In the few instances where in-context help is provided, the help text could be more specific.

Next steps

Adding in-context help text would make the search functions easier to use and improve the cognitive accessibility of the website.

Design Iterations

Home page

Since one of my main goals for the redesigned home page was to locate the most frequently-used actions at the top, my first prototype was actually of the information architecture and not the visual design. It was very helpful to have this already figured out when I started with the visual prototype!

The color scheme in my first visual prototype was eyedropped from the actual website, but it ended up making the design look muddled. I decided to work in grayscale instead and apply color once the values were right.

Search

I used the UT Libraries search results page as inspiration. Aside from changing the layout from horizontal to vertical, I also moved the search filters into a sidebar to distinguish them from the actual search results. Finally, I added in-context help buttons to the search fields and filters.

Top 100 (Browsing)

The original version of this page required so much vertical scrolling that it was difficult to navigate. Reorganizing the results into a table helped make better use of horizontal space while making it clear which section was what.

Reflections

This UI redesign project gave me many opportunities to improve my Figma prototyping skills, and I'm happy with how it came out overall.

One of the most important lessons I learned was to work in grayscale first (unless you're working with an established design system). Because I tried to add in color too early, I ended up putting in a lot of unnecessary effort trying to make the design work before I had really established the foundations. Taking a step back allowed me to see where the issues were, decide on the values I wanted, and create a final color scheme that reads well and is accessible. The US Web Design System's page on using color was a great resource.
